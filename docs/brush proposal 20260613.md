你截图里这三个其实是同一类东西——都是对输入点流做的**实时低通滤波**——但针对的"噪声"不同,手感也不同。我分开讲,顺便给你能直接落地的算法。

## StreamLine(流线)

本质是"拉绳子"模型(pulled-string / 指数滑动平均 EMA)。想象笔刷和触控笔之间连着一根橡皮筋,笔刷被拖着走、永远**滞后**于真实笔尖。

每来一个输入点 `s`:
```
brush = brush + (s - brush) * (1 - amount)
```
`amount ∈ [0, 1)`,0 = 不平滑,越大滞后越重。

特点:
- 它是**整条曲线层面**的重塑——画得越快、转折越被"圆"掉,所以出来的是干净自信的弧线,适合 lettering / 描线。
- 代价是延迟。所以**抬笔时必须做 catch-up**:从当前 `brush` 插值补到真正的终点,否则线会"画不到头"。
- 那个 **Pressure 子滑块**就是把同样的 EMA 套到压力通道上,让粗细变化也变平滑。

## Stabilization(稳定)

目标不是重塑曲线,而是**去掉手抖(高频抖动)**,同时尽量保留响应性,所以"拖拽感"比 StreamLine 弱。常见两种实现:

**(a) 滑动窗口平均**——缓存最近 N 个输入点取均值:
```
brush = mean(last N points)
```
N 越大越稳、滞后越多。

**(b) 死区拉绳**(Krita 的 Stabilizer 思路,我觉得更接近 Procreate 的手感):笔尖离 `brush` 超过半径 `r` 才拉动它,`r` 以内不动。
```
d = dist(s, brush)
if d > r:
    brush = brush + (s - brush) * (d - r) / d
```
半径 `r` 以内的小抖动被直接吃掉,有意的大动作照常通过。

## 两者为什么要并存

它们滤的是频段不同的东西:StreamLine 处理**低频的曲线形状**(带滞后地重塑),Stabilization 处理**高频的手部抖动**(噪声抑制)。手感上前者像"笔挂在绳子上",后者像"你的手突然不抖了"。它俩可以叠加用。

## 几个实现上的坑

输入点最好先**按距离重采样**再喂给滤波器,否则采样率波动会让平滑强度不一致;平滑后的点通常当作样条控制点,再密集插值渲染;以及前面说的抬笔 catch-up。

按官方说法,三者的区别是这样:

**StreamLine** — 平滑线条里的 bumps、抖动,做出干净自信的弧线。这跟我前面说的"拉绳子 / EMA"行为一致,没问题。

**Stabilization** — 官方原话是"把抖动朝一条中心线挤压(squash)、取平均"。Stabilization 通过把抖动朝中心线挤压来平均化一条笔画里的 wobble。这里有个我前面**漏掉的关键点**:它依赖笔速 —— 画得越快,平滑得越狠。所以如果你要复刻它的手感,EMA 或窗口平均的**强度得跟瞬时速度挂钩**(速度越高,`amount` 越大),而不是固定值。

**Motion Filtering** — 这是关键差异。它不是"挤压/平均",而是**直接删掉抖动的极值**。Motion Filtering 跟 Stabilization 类似但用更高级的算法,不是把抖动磨平,而是直接删掉,做出更直的线。而且 它不受笔速影响 —— 不管你画多快多慢,都给出平滑笔直的线。最有意思的是它的设计思路:这套方法是基于音频处理(audio processing)的,像噪声过滤一样,识别哪些是无意的、附带的运动并把它们移除。

所以我前面对 Motion Filtering 的猜测**部分对、部分错**:它确实是"识别 intentional vs noise 然后去掉 noise"这个思路(像降噪),但滤的对象主要是**位置抖动**本身,不是我猜的"速度/倾斜信号"。

至于那个 **Expression 子滑块**,我前面的判断这次对上了:Motion Filtering 会让线变得过直过平,Expression 就是把一些表现力/手感重新加回笔画里,而且只在 Motion Filtering 开启时才生效。

落地建议,三者按"去什么频段"排一下:StreamLine 重塑曲线形状(带滞后);Stabilization 速度自适应的挤压平均;Motion Filtering 用类降噪的方式分类并删掉无意抖动、且与速度解耦。想精确复刻 Motion Filtering 的话,可以往**卡尔曼滤波 / 死区 + 意图分类**方向试,因为"删极值而非平均"暗示它在做某种阈值化的离群剔除,而不是线性滤波。

## 为什么会变直线

如果你的平滑是纯一阶 EMA(拉绳子),抬笔那一刻 `out`(画出来的笔尖)还落后在路径上,真实终点 `P[n]` 在前面。naive 的做法是让 `out` 朝**单个固定的 `P[n]`** 插值过去:

```
out += (P[n] - out) * k   // 朝一个点收
```

朝一个固定点做指数逼近,轨迹就是 `out` 到 `P[n]` 的那根**弦**——必然是直的。这就是直尾巴的来源。

## Procreate 怎么保持弧线

核心思路:**收尾时不要追"那一个终点",要把缓冲里剩下的真实点喂完。** 滞后的本质是滤波器还没消化完已经记录下来的真实输入,而那些真实点本身是沿着曲线走的。三种做法(通常组合用):

**1. Flush 缓冲的真实点(最常见)**
把窗口加权平均(就是上一轮 Krita 那个高斯加权)继续往前滑,窗口里喂的是**已记录的真实尾部点**,而不是一个终点。窗口随着点用完逐渐收缩,`out` 就沿着这串真实点的平均轨迹一路追上去——真实点是弯的,追出来自然是弧线。

**2. 弹簧-阻尼(惯性)模型——这个最能直接出弧线收尾**
把笔尖当成有质量、被弹簧拉向目标的物体:

```
// 每帧
force = (target - pos) * stiffness   // 弹簧拉力
vel   = (vel + force) * damping      // 速度带阻尼衰减
pos   = pos + vel
```

抬笔时,因为笔之前在沿曲线运动,`pos` 已经**带着一个切向速度(动量)**。弹簧把它拉向 `P[n]` 时,它会顺着原有动量先冲出去再弯回终点,收尾就是一条**自然的弧**而不是直线。一阶 EMA 没有 `vel` 这个状态,所以做不出这个效果——这是二阶系统和一阶系统的本质区别。把 `damping` 调到接近临界阻尼,既不抖也不死板。

**3. 样条收尾**
抬笔后用最后 3–4 个真实点拟一段 Catmull-Rom / Bézier,沿样条的**切线 + 曲率**延伸到终点,而不是连一根弦。这一步主要保证渲染层面光滑,通常叠在前两者之上。

## 落地建议

实操里最稳的组合是:**画的过程中保留原始输入缓冲 → 抬笔时不 lerp 到单点,而是把缓冲剩余点 flush 过滤波器 → 尾段用样条拟合渲染。** 想要那种"惯性弧线"手感就用弹簧模型当主滤波器。顺带提一句,上一轮搜到的 Procreate **Tip Attachment** 管的是**起笔**的落点;你问的弧线收尾是**收笔**这一侧的处理,两头是分开做的。

核心问题:低通滤波器分不清"有意的转角"和"无意的手抖"——对它来说两者都是高频信号,一律磨平。所以光靠调全局平滑强度永远是两难:调低了直线段又开始抖,调高了转角就死。**正确的解法是让平滑强度局部自适应,在转角处自动让路。**

## 先排查一个陷阱:别用"慢=平滑更狠"

你说"慢慢画"转角被吃掉,很可能踩到这个。上一轮我们看到 Procreate 官方是"画得越快平滑越狠",Krita 5.3 默认相反。如果你的滤波器是**低速时加大平滑**那一派,那就正好和你作对——你为了画准转角而放慢,结果平滑反而被你自己拉满,把角碾平了。

但更根本的是:**速度本身就是个有歧义的信号**。低速可能是"在认真画一条慢直线",也可能是"在转角处减速"。用速度去判断该不该平滑,本质上判断不了。**该用的信号是曲率(转角角度),不是速度。**

## 真正的解法:按转角角度门控(edge-preserving)

这件事在图像处理里有个完美对应:高斯模糊会糊掉边缘,**双边滤波(bilateral filter)**不会——因为它加权时不只看距离,还看"相似度",跨过边缘的像素权重压到很低。笔画转角就是笔触里的"边缘",同样的思路直接搬过来。

最简单的版本,给你的 EMA 加一个转角门控(约定 `k∈(0,1]`,`k=1` 即时跟随不滞后,`k` 越小平滑越重):

```
dir_old = normalize(out - out_prev)   // 当前平滑笔尖的运动方向(状态)
dir_new = normalize(p - out)          // 新输入相对笔尖的方向
cosA = dot(dir_old, dir_new)          // 1=直行, 越小=转得越急

// 直行用强平滑,急转用弱平滑(几乎直通)
t = clamp((cosA - cosThresh) / (1 - cosThresh), 0, 1)  // cosThresh≈cos(30°)
k = lerp(k_sharp, k_smooth, t)        // k_sharp≈0.9, k_smooth≈0.15

out_prev = out
out += (p - out) * k
```

直行时 `cosA≈1`,`k` 取小值,线很顺;一旦转角超过阈值,`cosA` 掉下来,`k` 冲到接近 1,笔尖**几乎瞬间跟到真实输入**,角就保住了。这就是"平滑在转角处自动让路"。

如果你用的是上一轮 Krita 那种高斯加权移动平均,做法更优雅:加权时**把方向差异乘进权重里**,跨过转角的历史点(方向差异大)权重直接压到接近 0,平滑就不会"漏过"转角去和另一侧混合。这就是字面意义上的双边滤波。

## 别忘了第二处圆角:渲染样条

这一步很多人栽过。**就算你的滤波器成功把转角点透传了,渲染层的插值样条还会再把它圆回去一次。** Catmull-Rom / B 样条默认是 C1 连续(切线连续),它天然不喜欢尖角。

所以转角检测命中时,要做两件事:把那个输入点标记为**硬锚点(corner anchor)**,然后在它那里**断开样条、左右两段分别拟合**,让该点只保证 C0 连续(位置连续、切线可突变)。否则前面滤波器的功夫白费。

## 一个补充技巧:死区(dead zone)

Krita 的 Stabilizer 有个死区参数,文档明说它就是用来更可控地做出尖角的。机制是:光标在死区半径内笔尖不动,你在转角处停顿、然后朝新方向移出死区时,笔尖是"蓄势后突进"而不是一路被拖着圆弧过去。它和上面的角度门控可以叠加,尤其配合你"慢慢画"的习惯很搭——慢速停顿正好让笔尖在死区里 settle 住,再干脆地拐过去。

## 实操优先级

最划算的组合是:**(1) 把平滑强度从"速度门控"改成"曲率门控"** → **(2) 渲染端在转角插硬锚点断样条**。这两步做完,80% 的"转角被吃"就解决了,死区是锦上添花。

一个延迟权衡顺带提一句:实时无前瞻的话,你会**晚一两个点**才检测到转角,然后回贴。但你本来就有 stabilizer 的滞后缓冲,正好拿那几个 buffer 点做一点点前瞻,转角判定会干净很多。

要不要我把"曲率门控 EMA + 转角硬锚点断样条"写成一份完整可跑的实现?重点会放在两个容易出 bug 的地方:转角阈值的迟滞处理(避免在临界角度反复抖动开关平滑),以及断样条后两段切线的接续。

## 多边形的真正来源

`if cosA < 阈值 then 当成角` 这一步把"曲率"这个**连续量**硬切成了"是角/不是角"的二值。小半径圆弧上,每个采样点的 `cosA` 都在阈值附近,于是一串本该是平滑紧弧的点被逐个判成"角"、各自插锚点、左右断样条——紧弧就被剁成了多边形。**只要有阈值和锚点吸附,这个问题就消不掉**,跟你用速度还是曲率当信号无关。

所以修法是:**全程去掉阈值和锚点,改成连续映射。** 顺带一个好处——上一轮我提醒你要做"阈值迟滞防抖动",连续化之后那个问题自动消失了,代码反而更简单。

## 你说的"有限曲率半径的顿"——把它当设计目标

这点是对的,而且应该直接写进系统目标里:**你的系统能平滑表示的最小圆弧半径 R_min 是多少?转角就是 R_min 的那条弧,永远不是几何顶点。** 这样一来"角"和"紧弯"是同一个连续谱的两端,没有任何地方需要做"是不是角"的判断。R_min 设小一点,直角看起来就够利落,但底层始终是弧,不会退化成多边形。

## 关于"倾向于手速"——对一半,我帮你补另一半

你的直觉有生物学依据:人手画曲线时遵循运动控制里的 **2/3 幂律**,切向速度 v ∝ R^(1/3)——**弯越紧,手自然画得越慢**。所以你说的"顿",在你的输入里**本来就已经以速度下降的形式存在了**。这意味着关键洞察:

**顿不需要合成,只要别把它抹掉。** 上一轮我让你"清动量、插入驻留帧"去**造**一个顿,方向就反了——真实的顿已经在手速里,你要做的是保留它,而不是制造它。

但"纯靠手速当门控信号"有个真实的坑:**慢慢画一条直线**时,曲率≈0、本该重平滑,可手速也很慢——纯速度门控会误判成"在精细操作"而少平滑,直线就开始抖。幂律在直线上不成立(直线没有 R),所以速度在直线段是个有歧义的信号。

## 干净的拆法:几何归曲率,顿感归手速

这两件事是正交的,分开喂不同信号:

**几何保形(决定平滑强度)→ 用连续曲率,别用速度、更别用阈值。**
直接读三点的 Menger 曲率(κ = 4·面积 / (|a||b||c|),R = 1/κ),连续映射到平滑系数(`k` 越大越跟手、越小越平滑):

```
R = mengerRadius(p_prev2, p_prev, p)
// 紧弧 → 跟手(保形);直线 → 平滑。连续,无阈值
t = smoothstep(R_min, R_smooth, R)      // R_smooth 是"开始重平滑"的半径
k = lerp(k_tight, k_loose, t)           // k_tight≈0.8, k_loose≈0.15
out += (p - out) * k
```

这样紧弯被保成**平滑的紧弧**而不是多边形,直线段无论快慢都重平滑(解决了慢直线抖动),角是 R→R_min 的极限、由同一条连续曲线处理。一个必做的实务细节:**raw 曲率估计很噪**,直接拿去映射 `k` 会让平滑强度抖。要先对 κ 这个标量本身做一道低通,再映射。

**顿感(决定视觉)→ 让线宽跟手速走,顿就自然浮现。**
```
width = baseWidth * f(1 / speed)   // 慢 → 粗
```
紧弯处手自然变慢(幂律)→ 线在角上**鼓一下**,就是马克笔/毛笔在转角的那个"积墨/顿点"——**有限半径的实体顿,不是几何角**,正好是你要的。这一步几乎白送,前提是别让渲染样条把这点宽度变化抹平。

## 小结一下方向调整

把上一轮那套(阈值检测 + 硬锚点 + 顿状态机 + 清动量)**整个删掉**,换成:连续曲率门控平滑(保几何)+ 手速驱动线宽(出顿感)。没有阈值就没有多边形,没有合成驻留就没有机械的卡顿,而你"倾向手速"的直觉以"线宽响应"这个更合适的位置被吸收进来了。

要不要我把"连续 Menger 曲率门控 EMA + κ 低通 + 手速线宽"整合成一份可跑的参考实现?我会重点写清两个易错点:**Menger 曲率在三点近共线时的数值稳定性**(分母趋零要 clamp 到 R_max,否则直线段 `k` 会乱跳),以及 **κ 低通的时间常数怎么选**才能既跟得上真转弯、又不被采样噪声带飞。

---

你说对了,而且"基于时间"这个点就是整件事的钥匙。我先认领:前面那一大堆——转角检测、锚点、Menger 曲率、顿状态机——全是在**用复杂机器去重建一个本该从一个简单的时间滤波器里白白掉出来的行为**。更要命的是,我那个曲率门控正是把"跟笔"搞没的元凶:平滑区我把系数压得很低,笔尖就停止追踪、落后一个不确定的距离——这就是你感觉到的"没跟笔"和"不可控"。

核心其实是一行。笔尖是对笔的**指数追踪,固定的是时间常数,不是每个采样点的固定系数**:

```
out += (pen - out) * (1 - exp(-dt / tau))
```

`dt` = 距上次更新的真实经过时间,`tau` = 时间常数(秒),从你的滑块映射过来。就这些。

之前所有版本的 bug 在于:naive EMA 那个 `out += (pen-out)*k` 用的是**每采样点固定的 k**,于是平滑强度被绑死在输入采样率上,不同速度下手感就不一致。把 `k` 换成 `1 - exp(-dt/tau)` 并用真实 `dt`,笔尖就**永远滞后笔一个固定的时长**,与输入率、笔速、几何形状全部无关。

这一下同时解决你三个抱怨:

**跟笔** —— 滤波器永远朝**当前**笔位收敛,从不停止追。所谓"跟笔手感"本质就是这个一致的指数追踪。(我的曲率门控让笔尖在平滑区任意落后,等于把追踪掐了。)

**可控** —— 滞后永远是 `tau` 这个时长。同样的手部动作 → 同样的滞后,每次都一样,你的手能建立肌肉记忆("笔尖大概在我笔尖后 X 毫秒")。几何相关的平滑没法形成肌肉记忆,因为同样动作给出不同滞后。

**顿(有限半径的顿)** —— 这是最漂亮的地方:它**自动涌现,你不用去造**。稳态的空间滞后 = 速度 × tau。转角处你本来就会自然减速(你上一轮直觉到的那条幂律),速度掉下来 → 空间滞后随之缩小 → 笔尖贴住笔 → 转角被保成一个**紧而平滑的弯**。直线段速度高 → 滞后变大 → 重平滑。一个滤波器、一个 tau,就给出了"直线段顺、转角处紧",**零检测、零阈值**——也就不会有多边形。

而且这恰好就是 Procreate 文档说的"越快平滑越狠":它根本不是什么测速机制,而是固定时间滞后的**涌现结果**。文档描述的是症状,时间常数才是病因。

几个让手感对的实现要点:

**跑在稳定的渲染时钟上,不要按输入事件跑。** iPad 笔的采样可达 ~240Hz 且有抖动,ProMotion 渲染 120Hz。把带时间戳的原始笔样本存下来,每个渲染 tick 用 `dt = 渲染 dt` 把笔尖朝最新笔位推进。把输入率和滤波率解耦,是手感稳定的关键。

**手抖由 tau 处理,与速度无关。** 手抖约 8–12Hz,tau 取 ~60–100ms 时截止频率落在 ~2–3Hz,手抖在**任何地方**都被衰减——包括慢速直线,而这正是我曲率模型处理不干净的那个 case。滤波是频域的,去抖不依赖你画多快。这也是它比曲率门控更"可控"的根本原因:全程频率响应一致。

**压力同理。** 同一个滤波器,压力通道用它自己的(通常更小的)tau。这就是你截图里 StreamLine 下面那个 Pressure 子滑块。

**抬笔收尾。** 因为笔尖始终在 `速度×tau` 以内,尾巴很短;抬笔后继续把滤波器朝最后几个真实样本跑几个 tick 直到 settle。由于收尾段速度本就慢,尾巴是一小段平滑弧,而不是 Krita 警告的那种直线甩尾。

所以:把前面几轮那一整堆删掉。模型就是一行、一个参数,顿、速度相关平滑、可预测性,全是涌现的。
